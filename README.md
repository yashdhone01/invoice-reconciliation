Given a set of invoices and a set of incoming bank payments (both intentionally
messy — see [Assumptions](#assumptions--test-data)), it:

1. Matches each payment to the invoice it belongs to, using reference numbers
   first and falling back to customer + amount when references are unreliable.
2. Classifies every invoice as `Matched`, `Partially Matched`, `Overpaid`, or
   `Unmatched`.
3. Flags duplicate payments and orphan payments (money in with no matching invoice)
   for manual review.
4. Serves the results as JSON via `GET /summary`, with a plain-English summary
   generated alongside the numbers.

# Features

### Core functionality

- Generates realistic invoice and bank payment datasets
- Reconciles invoices against incoming payments
- Matches using:
  - Invoice reference
  - Customer
  - Amount (with configurable tolerance)
- Supports multiple payments against a single invoice
- Categorizes invoices as:
  - Matched
  - Partially Matched
  - Overpaid
  - Unmatched

### Exception detection

- Duplicate payments
- Orphan payments
- Missing payments
- Partial payments
- Overpayments

### AI Assistance

Produces a natural-language reconciliation summary alongside the structured
JSON response.

The reconciliation logic itself is fully deterministic. AI is used only to
summarize the results—not to make financial decisions.

### Dashboard

A lightweight dashboard built directly on top of the API displaying:

- KPI cards
- Status distribution
- Matched vs pending value
- Live exceptions table

No additional backend endpoints or database are required.

## Assumptions & test data

No input files were provided, so I generated my own — deliberately messy,
not a clean 1:1 dataset, so the matching logic actually has something to do:

- **Exact matches, partial payments, overpayments, duplicate payments,
  bank-charge rounding, garbled references, and a few orphan payments** are
  all represented in `data/invoices.csv` and `data/payments.csv`.
- **One invoice can receive multiple payments** (installments) — paid amounts
  are summed per invoice rather than assuming a strict 1:1 relationship.
- **A ₹10 tolerance** is applied by default to absorb bank charges/rounding —
  matching on exact paisa would be unrealistic for real bank data.

---

## Design decisions

**Matching order: reference number, then customer + amount fallback.**
Reference numbers are normalized (case/punctuation/spacing-insensitive,
matched on the invoice's numeric tail) since that's what a person actually
types into a bank transfer note. When that fails, falling back to customer +
amount mirrors how an accountant reconciles a messy statement by hand.

**Duplicate detection is a separate signal from match status.** An invoice
can be "Matched" on net amount while still being flagged as a duplicate-payment
risk (e.g. paid twice via two channels) — these need different follow-up
actions, so they aren't collapsed into one field.

**Dataclasses for the engine, Pydantic for the API boundary.** The matching
logic doesn't need validation overhead internally; Pydantic earns its keep at
the API layer, where it also generates the OpenAPI docs for free.

**Reconciliation runs once at startup**, not per-request — correct tradeoff
for a read-heavy prototype. A production version would re-run this on a
schedule or via an explicit `POST /reconcile` trigger.

**The AI summary degrades gracefully.** `ai_summary` in `/summary` is
template-generated by default so the endpoint stays fast and works with zero
setup; if an `ANTHROPIC_API_KEY` is present, it calls Claude for a genuinely
model-written narrative instead. A reviewer without a key shouldn't hit a
broken endpoint.

**Malformed data doesn't crash the app.** Invalid CSV rows are logged and
skipped in `main.py`'s loader, rather than taking down the whole service.

# Design Decisions

### Deterministic reconciliation

Invoice matching is deterministic and fully explainable.

The matching order is:

1. Invoice reference
2. Customer + amount fallback

Reference numbers are normalized before comparison to tolerate differences in
spacing, punctuation and casing.

---

### Payment tolerance

A configurable ±₹10 tolerance absorbs minor bank charges and rounding
differences that commonly occur in real payment systems.

---

### Duplicate payments

Duplicate detection is treated independently from reconciliation status.

An invoice may reconcile successfully while still being flagged as a potential
duplicate payment requiring manual verification.

---

### Multiple payments

Invoices may receive multiple payments.

Rather than assuming a strict one-to-one relationship, all matching payments
are aggregated before determining the reconciliation status.

---

### AI summary

The reconciliation engine remains deterministic.

AI is used only to convert structured reconciliation output into a concise,
human-readable narrative.

If an `ANTHROPIC_API_KEY` is available, Claude generates the summary.
Otherwise a deterministic template is used so the project runs without any
external dependencies.

---

### Dashboard architecture

The dashboard contains no business logic.

It consumes the same REST endpoints that any external client would use,
demphasizing API-first design and avoiding duplicated backend logic.

## What I'd do with more time

- Persist reconciliation runs to Postgres instead of in-memory/CSV, with a
  `POST /reconcile` endpoint to trigger re-runs on new uploads.
- A lightweight dashboard (Streamlit) visualizing matched vs. pending value over time.
- Fuzzy customer-name matching (`rapidfuzz`) for minor spelling variance
  between invoice and bank records.
- Indexing/batching for the matching step at real-world data volumes — current
  approach is fine at prototype scale, not at millions of rows.
=======
For production systems, payments would be indexed by normalized invoice
reference and customer to reduce matching toward **O(n + m)**.
